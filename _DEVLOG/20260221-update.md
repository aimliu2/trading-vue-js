# Revised Direction
- `TradingVue.js` is a large project that came with many issues.
- Scope of work should be clarified what it is and what it is not

### Scope of work
`TradingVue.js` should handle only these tasks;
- Visualize chart 
- Reactive, can be self-updated such as tick update, add indicator, add overlay, remove overlay
- Data willbe handled by external module

### Overview
- Core module is `Datacube.js` extended from 
    - `dc_core.js`, handling functions
    - `dc_event.js`, handling all events

### Current Issues
- some part of the coded needed debounce/throttle from lodash
- relied on expired dependency (moment.js)
- `Datacube` was a large object where each data parts were deeply watch by different components. Some module redundantly watched the same whole object. This caused performance issue when open multiple chart view.
- `dc_core.js` was coupled tightly with Vue2, using watch
- `dc_event.js` was listened to event from Vue2 componentes

### Optimiztion + upgrade to Vue3
- use `markraw`. see [markraw and uplifted state](./20260221-coderef.md#example-of-markraw--upliftcode)
- use `shallowRef`/`shallowReative` when component does not needed to rerender, then triggered when it have to.
- move throttle/debounce to utility function to handle rapid fired event i.e. move/ scroll
  - debounce : Handle rapid fire event. User stop the event. We are interested in last thing happen. **used it on resize, keytype input event**
  - throttle : Also Handle rapid fire event. This time user does not stop. function have to act on condition. **used it on infinite fetching, ** ensuring the original function is executed at most once within a specified time interval
- to prevent memory leak see [cleanup](./20260221-coderef.md#add-composable-to-prevent-memory-leak)

### Ai suggestion
- To efficiently manage a large data object with many child components responsible for both rendering and mutating specific parts, you should follow a strategy that prioritizes **localizing updates** and **minimizing re-renders.**

- Implement Granular Data Binding
Instead of passing the entire large object to every child, pass only the specific slice of data that the child needs.

- Mantain single source of truth
To maintain a single source of truth and prevent DOM-VDOM conflict by direct mutation, centralize the mutation logic in a parent or a dedicated state manager(i.e. pinia)
see [single source of truth](./20260221-coderef.md#single-source-of-truth-example-of-markraw--upliftcode)

- Always create a new object/array reference when updating data (using spread syntax or libraries like Immer) to ensure the framework can detect the change and trigger targeted re-renders

- To force component re-render Best Practice: use Key-Changing Technique
