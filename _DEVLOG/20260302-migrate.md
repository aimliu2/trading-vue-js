# Webpack → Vite Migration

## Overview
Migrated the bundler from Webpack (`webpack/dev.config.js`) to Vite (`vite.config.ts`).
The goal was to replicate all webpack settings faithfully while using Vite idioms.

---

## Step 1 — Fix alias paths and ESM imports
**File:** [vite.config.ts](../vite.config.ts)

- Removed `const path = require('path')` — `require` is not available in ESM modules
- Removed bare `__dirname` — not defined in ESM context
- All aliases now use `fileURLToPath(new URL('./...', import.meta.url))` anchored at the project root
- **Root cause:** `vite.config.ts` lives at the project root, but paths were copied from `webpack/dev.config.js` where `__dirname` pointed to the `webpack/` subfolder — the `..` was causing aliases to escape outside the project entirely

---

## Step 2 — Add `MOB_DEBUG` define and `server.host`
**File:** [vite.config.ts](../vite.config.ts)

- `webpack.DefinePlugin({ MOB_DEBUG })` → `define: { MOB_DEBUG: JSON.stringify(process.env.MOB_DEBUG) }`
- `devServer.host: '0.0.0.0'` → `server: { host: true }`

---

## Step 3 — Static file serving for `data/` and `assets/`
**File:** [vite.config.ts](../vite.config.ts)

- `devServer.contentBase: [../data, ../assets]` has no direct Vite equivalent (`publicDir` is single-dir only)
- Added inline `serveStaticDirs` plugin using `sirv` (already bundled with Vite) to serve both directories at the dev-server root

---

## Step 4 — Web Worker support
**Files:** [vite.config.ts](../vite.config.ts), [src/helpers/script_ww_api.js](../src/helpers/script_ww_api.js)

- Added `worker: { format: 'iife' }` — IIFE keeps the worker self-contained, matching webpack's `worker-loader` output format required by the blob-URL approach in `script_ww_api.js`
- Commented out `import {} from './script_ww.js'` in `script_ww_api.js`:
  - This was a webpack-only trick to trigger `worker-loader` to discover and bundle the file
  - In Vite it would bundle `script_ww.js` into the main thread and override `window.onmessage` (harmful side effect)

---

## Step 5 — Port `WWPlugin` as a custom Vite plugin
**File:** [vite.config.ts](../vite.config.ts)

`webpack/ww_plugin.js` worked by fetching the compiled worker from the live dev server (`http://localhost:{port}/script_ww.worker.js`) after emit. This pattern doesn't exist in Vite.

**New approach — `viteWWPlugin`:**
- Uses Vite's programmatic `build()` API to compile `script_ww.js` as an in-memory IIFE (no disk write)
- lz-compresses the output with `lz-string`
- Writes `src/helpers/tmp/ww$$$.json` — same output format as the original webpack plugin
- Runs in the `buildStart` hook so the JSON is ready before `script_ww_api.js` imports it
- Same change-guard as the original: only writes if the content changed

---

## Step 6 — Terser mangle reserved names
**File:** [vite.config.ts](../vite.config.ts)

- Vite defaults to `esbuild` for minification; switched to `minify: 'terser'` to unlock `terserOptions`
- `mangle.reserved: ['_id', '_tf']` — protects identifiers used by script std from being renamed
- **Requires:** `bun add -D terser`

---

## Step 7 — Fix `tsconfig.app.json` path aliases
**File:** [tsconfig.app.json](../tsconfig.app.json)

- Only `@/*` was mapped; TypeScript could not resolve the other 8 aliases
- Added all aliases to `paths` to match the Vite alias config:
  `@src`, `@data`, `@components`, `@component-js`, `@composables`, `@helpers`, `@icons`, `@overlay-comps`, `@stuff`

---

## Step 8 — Fix root `tsconfig.json` and `tsconfig.node.json`
**Files:** [tsconfig.json](../tsconfig.json), [tsconfig.node.json](../tsconfig.node.json)

- `tsconfig.node.json` extended `@tsconfig/node24/tsconfig.json` which was **not installed**
  - Replaced the `extends` with inline settings (`target: ES2022`, `lib: [ES2023]`, `module: ESNext`, `moduleResolution: Bundler`, `strict: true`, `skipLibCheck: true`)
- Root `tsconfig.json`: annotated clearly that it is a project-references compositor; webpack-era settings (`module: ES6`, `moduleResolution: node`) are kept for `ts-loader` compatibility but do not affect the Vite project references

---

## Remaining action
```sh
bun add -D terser
```
Required for `build.minify: 'terser'` (step 6) to work.
